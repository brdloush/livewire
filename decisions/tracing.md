# Decision: SQL Tracing Strategy

## The Problem
We need a way to trace SQL statements generated by Hibernate during an agentic or interactive REPL session (`trace-sql`). 
We want to avoid heavy external dependencies like P6Spy if possible, and rely on native Spring/JPA/Hibernate hooks.

A common approach is to use Hibernate's `StatementInspector` combined with a `ThreadLocal` to capture SQL only for the current execution context. However, `ThreadLocal` drops context across thread boundaries (e.g., when using `@Async`, `CompletableFuture`, or virtual threads). In these asynchronous scenarios, the spawned threads won't inherit the `ThreadLocal` state, meaning their SQL statements won't be captured.

## The Pragmatic Reality
In Spring and Hibernate, the `EntityManager`, JDBC `Connection`, and transactions are inherently thread-bound (via `TransactionSynchronizationManager`). When code spawns a background thread, that new thread generally has to create its own independent transaction and Hibernate session. Therefore, the vast majority (95%) of relevant data access happens synchronously on the main execution thread, making `ThreadLocal` perfectly adequate most of the time.

For the remaining 5%—when specifically debugging asynchronous flows—we can leverage the fact that Livewire is a **dev-only, single-user tool**. We do not need enterprise-grade, complex context propagation (which even tools like P6Spy struggle with without additional libraries). 

## The Solution: Dual-Mode Interception
We implement a tiny, ultra-fast Java state-holder (`LivewireSqlTracer`) that acts as a Hibernate `StatementInspector`. It contains *two* interception buckets:

1. **A `ThreadLocal` queue:** Clean, isolated, and ignores any background scheduled tasks the Spring app might be running.
2. **A `volatile` global queue:** Captures *every single SQL statement* fired anywhere on the JVM while active. Because Livewire is run locally in development, background database noise is usually minimal, allowing us to effortlessly capture all async/virtual thread database activity.

### Clojure API
In the `net.brdloush.livewire.trace` namespace, we expose two macros:
* `(trace-sql & body)`: Activates the thread-local queue. Use this for standard, synchronous debugging.
* `(trace-sql-global & body)`: Activates the global queue. Use this when you know your code spawns threads and you need to see the SQL they generate.

### Spring Integration
The `LivewireSqlTracer` is registered during application startup via a `HibernatePropertiesCustomizer` bean in the `LivewireAutoConfiguration` class. This cleanly drops our inspector into the Hibernate configuration properties without requiring the user to modify their existing application setup.
